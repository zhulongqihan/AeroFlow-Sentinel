# 扩展点、统一风格与面试包装

## 文档目标

这份文档解决三个问题：

1. 项目后续还可以往哪些方向扩展
2. 面试时应该如何统一叙事风格
3. 简历、项目介绍和面试追问应该怎么说

## 建议统一的项目表达风格

后续你无论写 README、简历、项目介绍还是面试回答，建议统一使用下面这套风格：

### 1. 先讲业务目标，再讲技术实现

推荐表达顺序：

1. 目标是什么
2. 为什么这个目标有业务价值
3. 我设计了什么系统
4. 关键技术难点是什么
5. 我做了哪些工程化落地

不要一上来就说“我用了 Spring AI 和 Milvus”。面试官更关心你解决了什么问题，以及为什么要这样设计。

### 2. 一直强调工程闭环

统一叙事主线：

1. 风险感知：告警和日志是输入
2. 推理分析：多 Agent 负责任务拆解与证据收集
3. 知识补全：RAG 负责提供内部文档依据
4. 结果交付：最终输出结构化 Markdown 报告

### 3. 统一项目关键词

建议长期固定使用这些关键词：

1. 航旅预订链路稳定性治理
2. Java 后端 + AI Agent
3. 多 Agent 编排
4. RAG 检索增强
5. SSE 流式交互
6. 低成本部署
7. 工程化落地

## 面试高频追问与回答思路

### 为什么要用多 Agent，而不是单 Agent

回答重点：

1. 单 Agent 容易把规划、执行、总结混在一起，长任务容易失控。
2. 多 Agent 可以把职责拆开，Planner 负责计划，Executor 负责证据执行，Supervisor 负责流程控制。
3. 这种设计更适合巡检类任务，因为它天然需要“先拆解，再验证，再汇总”。

### 你怎么控制工具调用失败和模型幻觉

回答重点：

1. 在 Prompt 中要求必须基于工具返回内容作答，禁止编造。
2. 对失败方向设置重试与终止条件，避免死循环。
3. 最终报告要求明确区分已验证证据和无法完成的部分。

### 你为什么要做 RAG fallback

回答重点：

1. 低配机器不一定适合完整拉起 Milvus。
2. 项目既要能展示能力，也要能稳定运行。
3. 本地 Markdown fallback 保证了“没有向量库也能演示知识增强能力”。

### 这个项目里最体现 Java 后端能力的部分是什么

回答重点：

1. 控制层、服务层、配置层和工具层划分清晰。
2. 用 REST 和 SSE 提供两套交互模式。
3. 配置开关控制 Milvus、MCP、mock 数据和 demo profile。
4. 文档上传、分片、向量化、入库和检索形成完整后端链路。

## 面试官最容易认可的技术点

1. 你不是只会调模型 API，而是做了完整 Java 工程。
2. 你能讲清楚为什么要有多 Agent 和 RAG。
3. 你考虑了低资源部署，不是理想化 Demo。
4. 你可以把业务场景、技术设计和工程实现讲成一个闭环故事。

## STAR 表达模板

### S：场景

航旅预订链路具有搜索、报价、下单、出票、退改签和外部 GDS 接口等多条关键路径，出现故障时往往需要同时查看告警、日志和内部处理手册。

### T：任务

我希望把这种复杂排障流程做成一个可交互、可展示、可扩展的 AI Agent 系统，并且能够运行在低配 ECS 上。

### A：行动

我基于 Spring Boot 与 Spring AI Alibaba 设计了双链路系统：普通问答链路使用 ReactAgent，多 Agent 巡检链路使用 Supervisor、Planner、Executor；同时集成 DashScope、Milvus、内部文档检索和 mock 风险证据，并通过 SSE 输出分析过程和报告。

### R：结果

最终项目具备了可演示的航旅稳定性分析能力，支持低成本部署和开源展示，也能较完整地体现我在 Java 后端、AI Agent 和工程落地方面的综合能力。
