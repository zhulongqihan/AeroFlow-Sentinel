# AeroFlow Sentinel 面试问答手册

## 文档目标

这份文档用于把项目材料进一步转化成可直接开口回答的面试内容，重点覆盖：

1. 自我介绍中的项目表达
2. Java 后端面试追问
3. AI Agent 方向面试追问
4. 项目难点、权衡和亮点

## 1 分钟项目介绍

“我做了一个叫 AeroFlow Sentinel 的项目，目标是把航旅预订链路里的稳定性巡检和知识问答做成一个工程化 AI Agent 系统。系统后端基于 Java 17 和 Spring Boot，普通问答部分使用 ReactAgent，多 Agent 巡检部分使用 Supervisor、Planner、Executor 协作完成任务拆解、证据收集和报告生成。知识层接了 DashScope Embedding 和 Milvus，同时为了适配低配 ECS，我还做了 Milvus 关闭后的本地 Markdown fallback。前端除了 SSE 流式展示结果，还专门做了一个值班驾驶舱，把服务端会话状态、一键演示场景和操作时间线直接展示出来。这个项目对我来说最大的价值，是把 Java 后端工程能力和 AI Agent 落地能力结合到了一个完整系统里。”

## 3 分钟展开版介绍

### 项目背景

航旅预订链路包含搜索、下单、出票、退改签和 GDS 接口等多个关键环节，一旦出现异常，排查过程通常要结合告警、日志和内部处理手册。

### 我解决的问题

我想做的不只是聊天问答，而是把风险巡检、证据查询、知识检索和报告生成形成闭环，让系统能辅助排查而不是只负责回答概念问题。

### 核心实现

1. 用 ChatController 和 ChatService 提供普通问答与 SSE 流式对话接口。
2. 用 AiOpsService 构建 Supervisor、Planner、Executor 三层 Agent 编排流程。
3. 用 QueryMetricsTools、QueryLogsTools 和 InternalDocsTools 提供监控、日志和知识证据。
4. 用 VectorIndexService、VectorSearchService 和 RagService 打通文档切片、向量化、检索增强链路。
5. 用 application-demo.yml 支持轻量部署，在没有 Milvus 时退化到本地 Markdown 检索。

### 项目价值

1. 体现了 Java 后端完整工程能力。
2. 体现了多 Agent 和 RAG 的实际落地能力。
3. 体现了低资源部署和真实展示意识。

## Java 后端方向高频追问

### 你这个项目的后端分层怎么设计的

回答思路：

1. controller 层负责 API 入口和响应组织。
2. service 层负责对话、多 Agent、RAG 和向量链路等核心业务。
3. agent/tool 层负责给模型暴露可执行工具。
4. config 层负责外部依赖和部署开关。
5. 这样的分层便于业务迁移和能力替换。

### 为什么你同时保留 REST 和 SSE

回答思路：

1. 普通问答适合 REST 完整返回。
2. 巡检报告和长推理过程更适合用 SSE 流式展示。
3. SSE 的好处是前端用户感知更好，也更符合 Agent 任务逐步执行的交互特征。

### 你怎么处理可选依赖，比如 Milvus

回答思路：

1. 通过配置开关和条件注解隔离依赖。
2. Milvus 关闭时不创建向量相关 Bean。
3. InternalDocsTools 自动切换到本地 Markdown fallback。
4. 这样既能保证功能演示，又能控制部署成本。

### 你觉得这个项目里最关键的后端设计点是什么

回答思路：

1. 不是单个接口，而是多条调用链都能贯通。
2. SSE、RAG、多 Agent 和配置降级可以同时成立。
3. 这说明系统不是一个临时 Demo，而是具备一定可演化性的后端工程。

## AI Agent 方向高频追问

### 为什么普通问答用 ReactAgent，巡检用多 Agent

回答思路：

1. 普通问答更偏单步工具选择，用 ReactAgent 足够。
2. 巡检任务涉及规划、执行和总结，天然更适合多 Agent。
3. 把不同任务复杂度映射到不同 Agent 架构，是更合理的工程选择。

### 多 Agent 架构为什么是 Supervisor、Planner、Executor

回答思路：

1. Supervisor 负责流程控制和阶段切换。
2. Planner 负责拆解任务和制定下一步动作。
3. Executor 负责真正执行工具并返回证据。
4. 这种分工有助于降低长任务中的混乱和幻觉风险。

### 你怎么确保模型不是瞎编结果

回答思路：

1. Prompt 明确要求必须基于工具结果作答。
2. 失败要如实返回，不能跳过。
3. 最终报告模板要求写清证据、根因和限制。
4. 这让报告更像“基于证据的分析”，而不是自由生成文本。

### 你为什么还要做 mock 数据

回答思路：

1. 项目需要支持低成本演示和开源展示。
2. 真实监控和日志系统不一定总能接入。
3. mock 数据能保证核心流程可演示，但架构上仍保留真实接入扩展位。

## 你可以主动强调的设计权衡

### 权衡一：完整能力和部署成本之间的平衡

我没有强行要求所有环境都必须拉起 Milvus，而是把“完整向量检索能力”和“轻量可演示部署”拆成两种模式，这能保证项目既能体现技术深度，也能真正部署出来。

### 权衡二：单 Agent 和多 Agent 的边界

不是所有任务都用多 Agent。问答链路用 ReactAgent 更轻，巡检链路才采用多 Agent，这样复杂度和收益更加匹配。

### 权衡三：真实系统接入和演示友好性

我保留了 Prometheus、CLS MCP 等扩展位，同时提供 mock 数据和 fallback 路径，使项目既能往真实生产方向演进，也能在求职展示阶段稳定运行。

## 最容易打动面试官的三个亮点

1. 你能从业务场景讲到技术设计，再讲到部署落地，而不是停留在模型调用层。
2. 你能解释为什么需要多 Agent、RAG 和降级策略。
3. 你能用值班驾驶舱直接给面试官看服务端记忆、一键演示和操作闭环，不需要靠口头脑补。

## 现场演示建议脚本

建议按下面顺序讲，成功率最高：

1. 先打开首页，指给面试官看值班驾驶舱能力标签，说明业务边界是航旅预订链路稳定性。
2. 再看服务端会话卡片，说明前端不是假状态，而是会主动查询后端会话沉淀情况。
3. 点击一个“一键演示场景”，让系统自动发起问答或巡检。
4. 演示时间线如何记录动作、模式切换和结果产出。
5. 最后进入回答详情，展开讲多 Agent、RAG、工具调用或降级方案。

这套顺序的优势是先把项目价值显性化，再展开技术实现，面试官更容易跟上。

## 如果面试官质疑“这是不是套壳项目”

你可以这样回答：

“我理解面试官担心的是很多 AI 项目只是接了一个模型接口。但我这个项目里，Java 后端主体、REST 和 SSE 接口、文件上传、文档切片、向量化、条件装配、多 Agent 编排和轻量部署策略都是自己搭出来的。模型当然是能力底座，但项目价值在于我把这些能力变成了一个完整系统，而不是只写了几个 Prompt。”

## 可以反问面试官的问题

1. 团队现在更关注 Java 后端工程能力，还是 AI Agent 落地能力的结合？
2. 你们在线上使用 AI 能力时，更看重问答助手还是流程自动化？
3. 团队在做智能化系统时，通常怎么处理模型效果和工程稳定性的平衡？

## 临场表达提醒

1. 先讲目标，再讲实现，不要先背技术名词。
2. 强调你做的是“工程化系统”，不是“单点模型调用”。
3. 主动承认边界，会显得更可信。
4. 多用“为什么这么设计”，少用“我用了什么框架”。
