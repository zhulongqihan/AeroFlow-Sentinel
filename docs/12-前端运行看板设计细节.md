# 前端运行看板设计细节

## 文档目标

这份文档是前端可视化增强的专项说明，专门回答下面几个问题：

1. 为什么要给首页增加运行看板
2. 看板里的每个模块分别对应什么技术能力
3. 它和后端会话持久化、SSE、多 Agent 巡检的关系是什么
4. 这块功能在系统展示里如何解释

## 为什么这个功能很重要

在很多 AI Agent 项目里，用户第一眼看到的是一个聊天框。这样有两个问题：

1. 业务边界不够清楚，容易被误解为通用问答 Demo。
2. 工程闭环不够直观，必须依赖你口头解释才能理解。

运行看板的目标就是把“业务语境”和“系统状态”前置到页面首屏，让系统在还没开始对话前，就已经能说明：

1. 我覆盖哪些航旅稳定性场景。
2. 我当前处于什么交互模式。
3. 我有没有服务端记忆能力。
4. 我能否快速触发标准化诊断。

## 运行看板的四层结构

### 第一层：业务能力标签

这一层解决“这是做什么的”问题。

固定显示的标签包括：

1. 搜索与报价异常。
2. 下单创建失败。
3. 支付与出票延迟。
4. 退改签履约积压。
5. GDS/供应商网关波动。

这些标签本质上是把领域建模结果映射到了 UI 首屏。

### 第二层：状态指标卡片

这一层解决“系统现在是什么状态”问题。

#### 卡片一：当前模式

显示 quick 或 stream，对应不同问答路径。

#### 卡片二：服务端会话

通过 GET /api/chat/session/{sessionId} 获取：

1. messagePairCount
2. createTime
3. updateTime

这张卡能直观展示：

1. 后端是否真的保存了上下文。
2. 当前会话已经累计了多少轮问答。
3. 最近一次更新时间是什么时候。

#### 卡片三：本地历史

这张卡显示 localStorage 里的历史会话数量，用来说明前端具备快速回看能力。

#### 卡片四：最近动作

显示最近一次关键用户动作或系统动作，例如：

1. 系统就绪。
2. 切换模式。
3. 发送提问。
4. 生成报告。

### 第三层：快捷诊断任务

这一层解决“怎么快速发起标准化诊断”问题。

它不是简单的快捷提问，而是一组预制的演示策略：

1. 新建会话，避免上下文污染。
2. 自动选择最合适的模式。
3. 注入预置 prompt，保证问题粒度稳定。
4. 自动发送请求，减少重复人工操作。

### 第四层：操作时间线

这一层解决“如何把过程串成处置闭环”问题。

时间线记录的是关键操作，不是所有细节日志。这样做的原因是：

1. 使用者需要的是可快速回看的流程摘要。
2. 页面展示需要突出处置轨迹，而不是淹没在调试细节里。

## 前后端数据边界

驾驶舱的数据来自两个层次：

### 前端本地状态

1. 当前模式。
2. 当前 sessionId。
3. localStorage 中的 chatHistories。
4. 当前页面的 activityFeed。

### 后端接口状态

1. /api/chat 和 /api/chat_stream 的执行结果。
2. /api/flight_guard 的流式巡检输出。
3. /api/chat/session/{sessionId} 的服务端会话信息。

这意味着运行看板不是完全前端自嗨，它至少有一部分核心状态是从服务端拉回来的。

## 为什么这个功能对系统展示有价值

### 1. 先把价值讲出来

过去你需要先提一串问题，再等回答回来，别人才会慢慢理解系统能做什么。现在首页就已经把场景、模式和会话能力展示出来了。

### 2. 操作路径更稳定

快捷诊断任务降低了自由发挥的风险，尤其适合时间有限的现场展示和快速复盘。

### 3. 更容易自然过渡到后端实现

你可以先从运行看板讲起，然后自然过渡到：

1. 为什么会话状态要查后端接口。
2. 为什么普通问答和巡检要分两条链路。
3. 为什么前端能直接承接 Markdown 和 SSE。

## 这块功能的边界与后续空间

当前版本仍然是“会话级运行看板”，还可以继续扩展为：

1. 巡检历史回放中心。
2. 风险场景配置中心。
3. 跨天巡检结果对比面板。
4. 告警趋势图和链路分段热力图。

“我这次不是单纯美化页面，而是把 Agent 系统里原本需要靠口头解释的能力，转成了首页可见的运行看板信息，让系统在首屏就能体现业务边界、状态沉淀和处置闭环。”
