# 面试演示脚本与观察点

## 文档目标

这份文档不是讲架构，而是讲“你现场怎么演示最有效”。它解决三个问题：

1. 面试时先点哪里
2. 每一步应该让面试官看什么
3. 如何把前端现象自然引到后端实现

## 适合的 5 分钟演示顺序

### 第 1 步：先展示首页，不急着输入问题

你先告诉面试官：

1. 这不是一个普通聊天框。
2. 首页顶部是值班驾驶舱。
3. 这里直接展示了业务场景、服务端会话、本地历史和一键演示入口。

这一句的意义，是先把项目定位定住。

### 第 2 步：指给面试官看能力标签

这一步重点让对方看到系统关注的是：

1. 搜索与报价。
2. 下单创建。
3. 支付与出票。
4. 退改签。
5. GDS/供应商网关。

这样能快速说明你做的是航旅稳定性治理，不是通用问答系统。

### 第 3 步：点开一个一键演示场景

推荐优先点：

1. 搜索报价波动诊断。
2. 下单创建失败排查。
3. 一键链路巡检。

原因是这三种最容易体现差异：

1. 快速问答。
2. 流式问答。
3. 多 Agent 巡检。

### 第 4 步：让面试官看服务端会话卡片

这是最值得强调的观察点。

你可以说：

“这里不是前端自己估计状态，而是会主动查询后端会话接口，显示当前累计了多少轮上下文和最近更新时间。”

这一句能把前端展示自然引到后端持久化实现。

### 第 5 步：看操作时间线

这一步重点不是 UI，而是闭环。

你可以让面试官看到：

1. 何时切换了模式。
2. 何时发送了提问。
3. 何时完成了回答或生成了巡检报告。

然后你再补一句：

“这个时间线是为了把用户动作、Agent 行为和系统反馈连接起来，现场更容易讲清楚。”

### 第 6 步：最后才展开技术解释

当面试官已经对页面现象建立认知后，再顺着讲：

1. 快速问答走 /api/chat。
2. 流式问答走 /api/chat_stream。
3. 一键巡检走 /api/flight_guard。
4. 会话状态来自 /api/chat/session/{sessionId}。

这时候对方理解成本会明显下降。

## 最推荐的讲解句式

### 开场一句

“我先不从代码讲起，我先给你看系统打开后的首屏，因为我希望这个项目一眼就能看出它不是普通聊天 Demo。”

### 讲服务端会话的一句

“这个卡片显示的是后端会话状态，不是前端假数据，所以它能证明系统具备服务端记忆和持久化能力。”

### 讲一键演示的一句

“为了降低现场演示的不确定性，我把典型场景做成了一键卡片，它会自动切模式并发起请求。”

### 讲多 Agent 的一句

“如果点一键巡检，走的就不再是普通问答，而是 Supervisor、Planner、Executor 的多 Agent 流程。”

## 面试官通常会注意什么

你要主动引导对方看这四个点：

1. 这是不是明确业务域的系统。
2. 前端展示有没有和后端真实能力关联起来。
3. 有没有可以稳定复现的演示路径。
4. 你能不能从页面现象讲回系统设计。

## 演示中的常见失误

### 失误一：一上来就点很多按钮

这样会让系统看起来混乱，也会打断你的讲述节奏。

### 失误二：一开始就讲太多框架名词

这会让面试官还没理解价值，就先被技术细节淹没。

### 失误三：只展示回答，不展示状态卡片

这样会浪费这次前端驾驶舱增强最有价值的部分。

## 最后一句怎么收尾

你可以这样说：

“我希望这个项目体现的不只是我会调模型，而是我能把 Java 后端、Agent 编排、会话持久化、可视化演示和低成本部署真正拼成一个完整系统。”
