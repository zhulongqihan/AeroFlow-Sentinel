# 前端结构与用户交互流

## 文档目标

这份文档专门解释前端为什么能表现为一个真实的值班工作台，而不是一个普通聊天框。读完后应该能回答：

1. 为什么当前前端仍坚持原生实现
2. 新增的运行看板到底展示了哪些信息
3. 快捷诊断任务、服务端会话状态和活动时间线分别解决什么问题
4. 前端如何和 REST、SSE、服务端会话接口协同工作

## 关键文件

1. src/main/resources/static/index.html
2. src/main/resources/static/app.js
3. src/main/resources/static/styles.css

## 这次前端增强的核心目标

前端原本已经能完成聊天、流式展示和巡检，但有一个明显问题：

1. 用户知道它能用。
2. 页面首屏缺少明确的业务语境和状态感。

所以这次增强不是换框架，而是补“可见能力层”，把系统亮点直接放到首页：

1. 运行看板：直接显示当前模式、服务端会话、本地历史和最近处置。
2. 快捷诊断任务：把典型航旅链路问题做成可点击卡片。
3. 活动时间线：把用户动作和系统反馈串成可回看的处置闭环。

## 页面结构

现在页面被拆成四块：

1. 左侧导航区：新建对话、历史会话列表。
2. 顶部运行看板：项目能力标签、状态卡片、任务卡片、操作时间线。
3. 主聊天区：普通问答、流式问答、链路巡检结果。
4. 输入区：消息输入、模式切换、工具菜单和文档上传。

## 运行看板包含什么

### 1. 能力标签区

这个区域固定展示项目覆盖的业务对象：

1. 搜索与报价异常。
2. 下单创建失败。
3. 支付与出票延迟。
4. 退改签履约积压。
5. GDS/供应商网关波动。

这样做的价值是：打开页面不用提问，使用者就能知道这不是泛聊天产品，而是一个明确面向航旅稳定性治理的 Agent。

### 2. 状态卡片区

这里有四张核心卡片：

1. 当前模式：显示当前是快速问答还是流式问答。
2. 服务端会话：通过 GET /api/chat/session/{sessionId} 查询当前会话在后端的沉淀状态。
3. 本地历史：显示浏览器 localStorage 中保存的会话数量。
4. 最近动作：显示用户最近一次操作和对应时间。

其中第二张卡片尤其关键，因为它把“前端界面”与“后端会话持久化”直接连起来了，能直观证明项目不是纯前端假状态。

### 3. 快捷诊断任务区

这里放了 5 个高频值班任务卡片：

1. 搜索报价波动诊断。
2. 下单创建失败排查。
3. 支付出票延迟分析。
4. 退改签积压处置。
5. 一键链路巡检。

每张卡片都带有：

1. 场景标题。
2. 演示说明。
3. 推荐的执行模式标签。

点击后前端会自动完成：

1. 新建会话。
2. 切换到合适模式。
3. 填入场景 prompt。
4. 对普通诊断任务，只做预填并等待用户确认发送。
5. 对链路巡检任务，才会直接触发巡检。

这解决了一个非常现实的问题：重复输入标准化诊断问题成本高，而且直接自动发送时容易带来等待时间过长或误触发。快捷任务保留了标准化入口，但把最终发送权交还给用户，诊断过程更稳定、更可控。

### 4. 最近操作时间线

前端维护一个 activityFeed，用于记录关键动作，例如：

1. 系统就绪。
2. 切换模式。
3. 发送提问。
4. 流式问答完成。
5. 巡检报告生成。
6. 文档上传成功。
7. 服务端会话同步成功。

这个时间线的价值不是“看起来酷”，而是帮助你现场讲清楚整个闭环：

1. 用户做了什么。
2. 前端如何响应。
3. 后端是否沉淀了会话。
4. 最终产出了什么结果。

## 前端主控制器做了什么

app.js 中的 AeroFlowSentinelApp 现在承担五类职责：

1. UI 初始化：绑定输入框、按钮、菜单、历史列表和驾驶舱卡片。
2. 状态管理：维护 sessionId、当前模式、是否流式执行、本地历史和活动时间线。
3. 接口协调：调用 /api/chat、/api/chat_stream、/api/flight_guard、/api/upload、/api/chat/session/{sessionId}。
4. 内容渲染：渲染 Markdown、代码高亮、SSE 增量内容。
5. 演示编排：通过一键场景把常见演示动作标准化。

## 用户动作是怎么流动的

### 流程一：点击一键演示问答卡片

```text
用户点击场景卡片
	-> 前端新建会话
	-> 自动切换 quick 或 stream 模式
	-> 自动填充预设 prompt
	-> 切换到工作台并聚焦输入框
	-> 用户确认后点击发送
	-> 命中 /api/chat 或 /api/chat_stream
	-> 回答返回后刷新服务端会话状态卡片
	-> 时间线记录“发送提问 / 完成回答 / 会话已同步”
```

### 流程二：点击一键链路巡检

```text
用户点击“一键链路巡检”
	-> 前端新建会话
	-> 直接触发 /api/flight_guard
	-> 后端执行多 Agent 编排
	-> 前端流式接收 Markdown 报告
	-> 时间线记录“发起巡检 / 生成报告”
```

### 流程三：普通手动提问

```text
用户输入问题
	-> 前端追加用户消息
	-> 根据模式选择 REST 或 SSE
	-> 回答渲染到聊天区
	-> 触发服务端会话同步查询
	-> 驾驶舱卡片更新会话轮数与更新时间
```

## 服务端会话状态是怎么打通的

这是这次增强里最关键的一个点。

前端不是自己猜测“现在有多少轮上下文”，而是在每次关键操作后调用：

```text
GET /api/chat/session/{sessionId}
```

后端返回：

1. sessionId
2. messagePairCount
3. createTime
4. updateTime

前端据此更新“服务端会话”卡片。这说明：

1. 左边历史列表是前端 localStorage 视角。
2. 驾驶舱里的服务端会话卡片是后端记忆视角。
3. 两者是两层不同但互补的状态。

## 为什么不改成 React/Vue

原因不是“不会”，而是当前目标不需要。

### 选择原生实现的理由

1. 这个项目前端核心任务是承载 Agent 能力展示，不是做复杂业务中台。
2. 页面交互虽多，但仍然集中在单页控制器模型里，复杂度可控。
3. 静态文件由 Spring Boot 直接托管，部署链路更简单，尤其适合低配 ECS。

### 这次增强仍然坚持原生实现的原因

1. 驾驶舱、时间线和演示卡片本质是状态映射，不需要引入更重的前端工程体系。
2. 保持单文件主控制器，便于完整讲清楚状态流转。
3. 低依赖意味着线上演示更稳，出问题面更小。

## SSE 与 Markdown 的配合方式

前端对 SSE 的处理方式没有变，但在“可见性”上更进一步：

1. 流式问答会在时间线上留下记录。
2. 巡检结果依然以 Markdown 输出，但现在可以结合驾驶舱状态一起讲。
3. 用户能够从页面看到“当前系统正在做什么”，而不是只看到一段最终文本。

## 这次增强的展示逻辑

推荐按下面顺序理解或展示页面：

1. 先看顶部能力标签，确认系统边界。
2. 再看服务端会话卡片，理解页面状态和后端记忆的关系。
3. 点击一个快捷任务，让系统自动进入诊断流程。
4. 观察时间线如何记录模式切换、提问、报告生成。
5. 最后再进入消息详情，看后端 Agent、RAG 或工具调用输出。

## 当前边界

1. 驾驶舱的状态仍然是“单会话级可视化”，不是全局运维大盘。
2. 还没有把历史巡检结果做成跨天对比视图。
3. 一键演示场景当前是静态预置，后续可以配置化。

## 文档化表达建议

可以这样概括这次前端增强：

“我没有单纯做一个聊天页面，而是给 Agent 系统补了一层业务运行看板，把业务能力、服务端记忆状态、快捷诊断任务和操作时间线都放到了首页，让系统在首屏就具备场景感和处置感。”
